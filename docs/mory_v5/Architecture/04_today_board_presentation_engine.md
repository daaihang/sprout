# 04. Today Board Presentation Engine

## 1. Goal

Today Board should be generated by explainable rules and rendered with type-specific presentation components.

## 2. Inputs

Inputs to the board engine:

- Memories.
- Graph context.
- Record analysis index.
- Pipeline statuses.
- Local preferences.
- Current date.
- Limit.

## 3. Candidate Model

Each candidate should include:

- Stable card key.
- Card kind.
- Target type.
- Target ID.
- Source record IDs.
- Render value.
- Priority.
- Reason.
- Created at.
- Updated at.
- Layout hints.

## 4. Preference Application

Preferences are applied after candidate generation:

- Hidden/dismissed candidates are removed.
- Pinned candidates sort first.
- Preference updatedAt affects snapshot updatedAt.
- Invalid target preferences are ignored.

Preference fields:

- `syncKey`
- `schemaVersion`
- `boardKey`
- `cardKey`
- `cardKind`
- `targetType`
- `targetID`
- `isPinned`
- `isHidden`
- `dismissedAt`
- `updatedAt`

## 5. Rendering

Renderer maps card kind to component:

- MemoryBoardCard.
- StorylineBoardCard.
- ReflectionBoardCard.
- SystemPromptBoardCard.
- ContextClusterBoardCard.
- PendingActionBoardCard.

Shared card wrapper handles:

- Padding.
- Background.
- Border.
- Menu.
- Pin indicator.
- Tap action.

## 6. Debug Snapshot

Internal debug snapshot should include:

- Generated date.
- Limit.
- Input counts.
- Preference counts.
- Final items.
- Per-item priority/reason/source IDs.

## 7. Tests

Required tests:

- Empty board creates system prompt.
- Today memories rank above old memories.
- Context/graph/salience boost priority.
- Accepted recent storylines appear.
- Suggested reflections are limited.
- Saved reflections do not appear as suggestions.
- Pin/hide/dismiss persists.
- Debug snapshot exposes reasons.

## 8. Acceptance Criteria

- Product UI and debug use same board engine.
- Card reasons are non-empty.
- Preferences persist locally.
- Deleted targets do not render.
- Board rendering has type-specific cards.
