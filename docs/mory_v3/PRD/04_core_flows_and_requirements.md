# 04. Core Flows And Requirements

## 1. 产品的第一原则

Mory 的核心体验不是“页面多”，而是四条闭环是否成立：

1. `Capture`
2. `Organize`
3. `Retrieve`
4. `Reflect`

只要这四条闭环成立，产品会越来越强。  
如果只继续加卡片和功能，而这四条闭环没有强化，产品只会更复杂。

## 2. Capture 流

### 2.1 目标

让用户以低摩擦方式把生活材料快速带入系统。

### 2.2 支持的输入单位

输入单位不再是“必须写一条完整日记”，而是：

- `text`
- `photo`
- `audio`
- `music`
- `location`
- `weather`
- `link`
- `todo`
- `document`

补充边界：

- `person`
- `mood`
- `decision fragment`

可以是 capture 时的显式输入或后续分析语义，但不是一级 `ArtifactKind`。

### 2.3 产品要求

1. 用户应能从首页直接发起 capture
2. capture 可以是单一 artifact，也可以是一次 record 聚合
3. 输入后应立即落本地，不依赖服务端成功才完成保存
4. 用户显式填写的字段优先级高于 AI 推断
5. capture 后应能被快速看见，而不是“存进黑箱”

### 2.4 体验要求

- 文字输入不能重
- 图片、地点、人物添加要短路径
- 语音输入不应被设计成 gimmick
- 输入完成后要尽快生成可视结果

## 3. Organize 流

### 3.1 目标

让用户的材料不仅被保存，而且被组织。

### 3.2 组织方式

Mory 的组织方式必须是混合型，而不是只靠时间轴：

- 时间组织
- 空间组织
- 人物组织
- 主题组织
- 阶段组织

### 3.3 产品要求

1. 首页空间布局必须可持续保存和复现
2. 人物、地点、决定等对象要能成为独立入口
3. 不同 artifact 之间需要支持被关联和聚合
4. 用户可以接受系统默认组织，也可以逐步手动调整

### 3.4 不该做成什么

- 不要要求用户手工维护复杂文件夹结构
- 不要把布局编辑做成复杂设计器
- 不要让“整理”变成负担大于收益

## 4. Retrieve 流

### 4.1 目标

让用户以后真的能找回来。

### 4.2 检索维度

必须逐步支持：

- 按时间
- 按人物
- 按地点
- 按主题
- 按阶段
- 按类型
- 按情绪

### 4.3 产品要求

1. 用户在人物页中可以看到与该人的相关记忆集合
2. 用户在记录流中可以做精确时间回看
3. 用户在搜索中不只看到文本匹配，还能看到相关对象
4. 系统应具备主动回看能力，如同日往年、近期关系回顾、阶段回顾

## 5. Reflect 流

### 5.1 目标

让系统不是“记事本”，而是“有意义的记忆系统”。

### 5.2 Reflection 的触发

反思不应到处泛滥，而应在这些场景出现：

- 新记录完成后，做轻量反思
- 某主题反复出现后，做模式反思
- 某关系积累到一定密度后，做关系反思
- 某阶段结束或拉长后，做阶段反思
- 用户主动请求复盘时，做深度反思

### 5.3 产品要求

1. 反思输出必须能解释“为什么得出这个结论”
2. 反思应引用已有材料，而不是凭空说教
3. 高成本 AI 反思不应该成为每次保存的必经步骤
4. 用户应能对反思进行忽略、修正、收藏、展开

## 6. 核心页面要求

### 6.1 Home

职责：

- capture 入口
- 今日/当前记忆空间
- 可浏览的 composition
- 主动回看入口

关键要求：

- 快速加载
- 卡片/元素适配不同内容类型
- 支持多条记录聚合视觉化

### 6.2 Record Detail

职责：

- 显示一次 capture 的完整上下文
- 允许用户纠错、补充、编辑关联对象
- 展示该记录的分析快照

### 6.3 People

职责：

- 关系记忆入口
- 该人物相关的 artifacts、records、themes、reflections

### 6.4 Arc / Period

职责：

- 阶段性回看
- 长期模式展示
- AI 深度反思入口

### 6.5 Search

职责：

- 统一检索入口
- 把全文检索升级成“对象 + 关系 + 阶段”检索

## 7. 功能优先级

### 7.1 P0

- 快速 capture
- 本地持久化
- 首页 composition 化
- 人物基础入口
- 记录详情
- 轻量 AI 分析

### 7.2 P1

- Artifact 统一模型
- 语义检索
- Entity graph 基础层
- Reflection snapshot 持久化
- 主动回看入口

### 7.3 P2

- Temporal arcs
- 深度反思
- 跨阶段模式发现
- 高级导出与迁移

## 8. 非功能需求

### 8.1 隐私

- 默认本地优先
- AI 上行数据最小化
- 让用户明白哪些数据被上传分析

### 8.2 性能

- 首页滚动要稳定
- 本地存储迁移可控
- 分析链路异步不阻塞 capture

### 8.3 可演进性

- 新内容类型应通过 artifact 扩展，而不是新建一整套平行系统
- UI 改版不应迫使底层对象重建

### 8.4 可解释性

- AI 输出可溯源
- 关系与阶段形成应可解释

## 9. 明确禁止的方向

为了防止产品失控，以下方向在现阶段应明确不做或谨慎做：

- 无边界扩展卡片种类
- 聊天机器人式主交互
- 重度社交分享
- 多端协作和公共 workspace
- 让 AI 覆盖用户显式填写事实
