# 产品现状总览

## Mory 现在是什么

Mory 是个人知识管理、个人记忆增强和 AI 反思类应用。它的目标不是只保存一条条记录，而是把用户的文字、照片、语音、地点、音乐、手记建议、情绪和人物关系连接起来，形成长期可用的个人记忆系统。

现在的 Mory 已经具备四层能力：

1. 快速记录生活片段。
2. 保存记录背后的上下文。
3. 按用户排布复现这些记录卡片。
4. 用 AI 和本地图谱把片段变成长期线索。

## 当前已经跑通的主链路

用户可以在 App 内新建记忆，也可以从分享、App Intent 或 Apple Journaling Suggestions 导入内容。保存时，Mory 会先把 `RecordShell + Artifact[] + ArtifactSemanticDigest[] + MemoryCardArrangement + AffectSnapshot[]` 写入本地。默认状态是 `.notScheduled`，也就是先保存事实，不等于已经启动 AI 分析。

只有显式触发分析时，Mory 才会把这条记忆和已有的个人档案、人物、历史片段、情绪、地点、Arc、Reflection 等上下文一起组织起来，然后走 Analysis 链路。

分析结果不会直接把所有 AI 判断变成事实。更安全的部分会写入本地分析结果和图谱；高风险内容会变成 proposal 或问题，等待用户确认。

## 当前最大的问题

功能已经很多，但“用户和产品负责人怎么知道它工作到哪一步”还不够清楚。

典型问题包括：

- 保存后 AI 是否分析完成，用户不容易一眼看见。
- Apple Journaling Suggestions 导入的内容，有些来源和证据不够明显。
- Share 进入 Mory 后，成功、失败、恢复路径还需要更明确。
- 人物画像、Self Profile、GraphDelta proposal 的证据不够面向普通用户。
- 未来订阅和额度限制还没有正式产品化。

## 当前状态判断

| 模块 | 状态 | 人话解释 |
| --- | --- | --- |
| App 内新建记忆 | 可用 | 主流程能用，仍需更清楚展示来源和分析状态。 |
| 文本记录 | 稳定 | 是最可靠的基础输入。 |
| 图片记录 | 可用 | 能保存和参与分析，图片理解细节仍需增强。 |
| 视频记录 | 已接入 | 能作为媒体保存，记录卡片已有预览；播放器和分析还不够完整。 |
| 语音记录 | 已接入 | 能转写和生成内容，但需要防止 AI 润色覆盖用户编辑。 |
| Apple Journaling Suggestions | 已接入 | 能导入系统建议，Live Photo 保持单个 artifact；多类型展示和证据可见性还需完善。 |
| Share / External Capture | 已接入 | 能进入统一记录链路，但真机体验还要继续验收。 |
| Mood / Affect | 已接入 | 已经不是单个心情字符串，但用户可见的情绪历史还不足。 |
| 卡片排布和复现 | 可用 | 已切到固定列宽瀑布流、卡片自适应高度、contentDensity、stack/group、贴纸挂点 schema、rotation/nudge/zIndex、详情复现和 Card Debug 验收；正式 UI 拖拽和视觉 polish 还要继续。 |
| Self / People / Graph | 已接入 | 底层结构存在，产品里的“我”和人物画像仍需清晰页面。 |
| Analysis | 可用 | 已是新记忆分析主链路，但状态展示和失败恢复还需增强。 |
| 通知和后台 | 已接入 | 有基础机制，仍需真机长测和节奏设计。 |
| 订阅 / 付费墙 | 只有底层 | 还没有产品入口和正式截断策略。 |

## 现阶段目标

下一阶段不应该继续盲目堆功能，而应该先让 Mory 变得“可理解、可观察、可验证”：

- 用户知道自己保存了什么。
- 用户知道 AI 什么时候在工作。
- 用户知道哪些结论是 Mory 推断的，哪些是自己确认的。
- 产品负责人知道每个功能现在处于哪个状态。
- 未来加订阅、额度、隐私策略时，不需要重新猜数据链路。
