# Mory 产品系统手册

这是一份给人看的 Mory 总手册。它不要求读者懂 Swift、Go、SwiftData 或 API 合同；它回答的是产品层面的几个问题：

- Mory 现在到底能做什么？
- 用户从哪里进入，每一步会发生什么？
- 数据会怎样保存、分析、展示？
- AI 在什么时候介入，会不会挡住用户？
- 哪些能力已经可用，哪些只是接上了但还不稳定？
- 未来做订阅、付费墙、额度限制时，应该截断哪些功能？

更详细的技术事实在同级目录 [mory_system_handbook](../mory_system_handbook/README.md)。这里是面向产品、设计、测试、运营和非专业开发协作者的中文版本。

## 当前一句话结论

Mory 已经不是单纯的笔记或日记 App。它现在是一套“记录生活片段 -> 保存多模态上下文 -> 生成长期个人记忆结构 -> 等用户确认后逐步沉淀”的系统。

但它还没有完全达到发布级体验。核心问题不是“没有功能”，而是“功能太多但状态不够透明”：用户不一定知道 AI 有没有分析完、哪些内容来自手记建议、哪些人物画像是 AI 猜的、哪些纠错会影响后续分析。

## 状态词说明

| 状态 | 给人的解释 |
| --- | --- |
| 未开始 | 还没有真实实现。 |
| 只有底层 | 数据模型、服务或接口有了，但用户基本看不到。 |
| 只有调试 | 开发者能在 Debug 页面看到，但普通用户不能自然使用。 |
| 已接入 | 入口和数据链路存在，但体验或边界情况还不完整。 |
| 可用 | 用户能完成主要流程，但仍有已知限制。 |
| 稳定 | 主流程、失败处理、状态显示和测试都比较可靠。 |
| 打磨完成 | 接近正式发布质量，包括文案、视觉、无障碍、监控和边界体验。 |

## 以后怎么用这份手册

每加一个新功能，都要先回答这几个问题：

- 用户在哪里看到它？
- 用户做完动作后，数据保存在哪里？
- 是否调用 AI？什么时候调用？失败会怎样？
- 这个功能能否被免费版使用？如果不能，在哪里截断？
- Debug 和用户界面分别能看到什么状态？

如果这些问题答不清，就说明功能还没真正进入可控状态。
